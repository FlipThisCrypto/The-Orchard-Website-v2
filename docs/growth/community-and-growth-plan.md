# Community & Growth Plan

> Canonical growth doc. Integrated by the Lead from Grok's **ORCH-031**
> (`council/grok/submissions/ORCH-031-v1.md`). Date: 2026-06-17.
> Built on the three pillars (`positioning-and-messaging.md`) and the ratified positioning (ADR-0003).

Realistic for a solo founder with a small early community (~3 Trees). Growth = **more real Trees
producing verifiable data, more people who can inspect/extend the stack, visible proof the network
is alive and forkable** — not vanity metrics. Anti-hype; honest about stage.

## Channels (order of effort for a solo founder: GitHub > X light > targeted Reddit > product surfaces > Discord)
1. **X (primary public channel)** — already referenced on the live site; ship updates, short operator stories, educational threads reinforcing "verifiable"/"forkable." Establish a consistent handle + rhythm before the globe launch.
2. **GitHub (the technical core, highest for participation)** — issues ("tell us what breaks"), PRs for drivers/examples, Discussions, stars as a quiet signal. All activity is verifiable by nature.
3. **Reddit (targeted, not broadcast)** — r/chia (Chia-native) + r/esp32, r/arduino, r/raspberry_pi (makers who run sensors). High-quality posts only on real progress; cross-link GitHub + site.
4. **Direct product surfaces (the on-ramps)** — flasher, claim, MintGarden Passes, eventually the globe. Make every step feel open and supported; "report issues on GitHub" + "share your Tree."
5. **Discord (light / seeded later)** — first engage existing Chia Discords; create a dedicated server only after sustained real usage (multiple active operators).
6. **Maker / ESP32 / environmental communities** — Hackaday, project write-ups; share concrete forkable artifacts (driver, wiring, enclosure).

## Content cadence (sustainable for one person)
Substance over volume. **X: 2–4 items/week max** — one ship/status update when something verifiable
moves; one "how it actually works" thread mapped to a pillar; an operator/tester highlight;
consistent replies to real planters/questions. **GitHub:** respond to every issue, review PRs
promptly. **Rule:** never post just to post; transparency about delays builds trust.

## Community mechanics (tied to real, verifiable participation — no fake hype)
- **Orchard Pass = core identity** (Genesis Passes live; ORCH-0001 first). Visible credential; future perks (globe visibility, early access to new sensor classes, recognition) without promising token multipliers.
- **Verifiable participation:** public "running Trees" highlights from attested Seasons; credit for merged drivers/docs; "First Grove" recognition; light referral (public thanks for helping someone flash+claim).
- **Quests / ambassador (very light, voluntary):** "deploy + prove one full Season with a new sensor type, share Explorer data + GitHub link"; reliable testers named in release notes. Real infrastructure, not gamified busywork.

## Globe launch hooks
1. **"See your real Tree on the globe" (primary)** — every claimed Tree visualized with fruit from real readings. *"Your Tree is already on the map — claim it, watch the Seasons and fruit update."* (Pillars 1 & 2.)
2. **"Real deployments, real density" reveal** — launch shows exactly the live Trees (starting with KY); plant before/after launch and you appear. (Anti-hype honesty; "See the network.")
3. **"Fork or extend the view"** — the globe/visualization is open to extend; *"the same way you add a sensor driver, you can add how a new data type appears as fruit — early contributors get featured."* (Pillar 3 + makers.)
